# Results — SNG-044 (Item relevance + bonus-count cap)

Date: 2026-07-11 · Shipped v1.8.10 · 723 smoke checks (+4) + parse_probe green. Status: **shipped, awaiting Erik preview-legs before complete_pending_review** (Part A). Part B is Aevi's content audit. Only Aevi closes.

**The "hoarding broad-tagged items stacks bonuses" fix.** `equipmentBonus` summed +N over EVERY owned item whose `bonusTags` intersected the action, capped only on the total — so a bag of broadly-tagged items (the Traveler's Pack especially) helped nearly everywhere and stacked.

## Part A — cap the contributor count (engine, CCode)
- `equipmentBonus` now takes the **best-matching item only** — the top `equipmentBonusTopN` contributors (default **1**), sorted by each item's bonus. The total `equipmentBonusCap` stays a backstop.
- **Principle:** the right tool helps; a bag of tools does not help more.
- `helpers` names the item(s) that actually aided the roll — and it already flows to the roll receipt via `resolution.equipHelpers`, so the player sees "your climbing rope helped here," not an opaque +N.
- `equipmentBonusTopN` is tunable (a deliberate "two tools" rule is one line of content) but defaults to strict best-of-one.

## Guardrails held
Engine owns the count-cap + total-cap; content owns relevance. Reused the existing `bonusTags`/`equipmentBonus` seam. **Backward-safe** — best-of is a strict narrowing, no schema change; single-item situations are unchanged (a lone matching tool still gives its full bonus). Suites + parse_probe green. Never touched the ErikIAm pipeline.

## Part B — Aevi content audit (owed)
