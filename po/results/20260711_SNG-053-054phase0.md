## SNG-054 Phase 0 — interim skill-graph zoom/pan ("do NOW; Erik can't use the screen")
`renderSkillGraph` emitted a fixed-viewBox SVG that overflowed unusably on desktop. Pure viewport work on the existing render (NOT the Phase 2 redesign):
- The graph content is wrapped in a `<g class="graph-vp">` transform group inside a fixed-height, `overflow:hidden` container.
- **Wheel-zoom** toward the cursor, **drag-pan** (mouse + touch), **+ / − buttons**, and **⤢ fit-to-view** (resets to identity).
- Zoom/pan **state persists across node-select re-renders** (`graphView`) so a click doesn't reset the view; a drag is distinguished from a node-select (a moved-drag suppresses the select).
- Verified live: two + zooms → `scale(1.5625)` panned to keep the cursor point fixed; **⤢** resets to `scale(1)`.

## SNG-053 — portrait fidelity (form leads) + image lightbox
An Ent rendered as a plain human. Root cause: `characterPromptSeed` LED with the literal `"character portrait"` (biasing the model to a human) and the character schema had **no form field**, so non-human physiology never reached the image.
- **`formOf()` + FORM-led prompt (`art.js`):** the character/NPC prompt now **opens with the physical form** (species/lineage/appearance) — e.g. `"a towering treefolk of bark and heartwood, moss-bearded … full-body character portrait"` — then culture/gear/arc. `assembleImagePrompt('npc')` leads with the NPC's form too, so a generated non-human NPC renders non-human. Human is a **stated** default (`"a person"`), never the unspoken assumption.
- **`character.form` field + a "✎ Appearance" editor** on the character sheet — Erik describes the form; it's stored, **shown on the sheet**, and the portrait is **re-minted** to match. Additive/backward-safe.
- **Image lightbox:** any portrait / NPC / location / moment / gallery image is clickable → a larger modal (Esc / click-backdrop / ✕ dismiss; **← / →** arrow-through the gallery). One delegated handler, wired once.
- The deterministic seed, `SAFETY_TAIL`, and the rating/minor floors are unchanged (the form text still routes through `sanitizeImagePrompt`).

## 🐞 Boot bug found + fixed (in-browser verification earned its keep)
The first fresh-port load hung on "Loading the Valley…". `_lightboxWired` (a `let`) was declared *after* the `boot()` IIFE that calls `wireLightbox()`, so `wireLightbox` hit it in the **temporal dead zone** → boot threw before rendering. `parse_probe` never reaches boot (it throws earlier on a top-level `document.getElementById`), so it couldn't catch this. Fix: moved the declaration to the module-top `let` block. This is the exact class of bug the "live-verify on a fresh port" discipline exists to catch.
