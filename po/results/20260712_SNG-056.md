# Results — SNG-056 Location header desync

Date: 2026-07-12 · v1.8.27 · npm test green · boots clean on a fresh port. Status: **shipped, awaiting Erik leg.** Only Aevi closes.

## What was actually wrong (substrate finding)
The spec assumed the header rendered from a "cached or build-time string." **It does not** — the play header (`renderPlay` → `location-tag`) already renders from `hereNow()` = `CONTENT.locations[character.currentLocationId]`, the authoritative source, and so do the map "you are here" and the GM context. The header was **faithfully showing a stale truth:** the GM had moved the character in FICTION (a travel montage to the Disputed Zone) and there was **no op to update `currentLocationId`**, so the authoritative location itself never changed. Auditing the surfaces confirmed none held a stale cached name — the missing piece was upstream.

## The fix — a GM `moveTo` op
- **`gm.js`:** contract adds `moveTo{location, why}` — "THE HEADER FOLLOWS THE FICTION: whenever your narration moves the character to a different established place, emit moveTo so world state and the header agree." Added to the salvage whitelist.
- **`state.js`:** new pure `resolveLocationId(ref, locations)` — resolves an id or a place NAME (exact id → slug → exact name → loose name) to a real loaded location id, or null.
- **`app.js` `applyTurn`:** reads `turn.moveTo`, resolves it, and if it names a real place other than the current one, updates `currentLocationId` + `notePlaceVisit` + perception + image — *before* the `activeScene` anchor is set, so the anchor, header, map, and GM context all follow. An unknown ref is ignored (the engine only honors a move to a place that exists).

## Verification
- `npm test` green — 5 new resolver checks (exact id, exact name, loose fragment, non-existent → null, empty → null).
- Fresh-port boot clean (new import + dispatch, no regression).
- **Erik leg (needs a live GM):** "Travel/be-led somewhere — verify the header, the sheet's place, and the map all say where you actually are, immediately." The engine path is proven; the GM emitting `moveTo` on a fiction-move is the live behavior to confirm.

## spec_boundary
The spec scoped this as "point all surfaces at the one source + a display fix." The surfaces were already correct; the real fix is the **new `moveTo` op** so the source itself stays true when the GM moves the player narratively. Reported rather than forcing the display-only framing.
