# Results — SNG-085 (Repair panel + GM "do the doable part")

Date: 2026-07-12 · v1.8.43 · npm test green (smoke + parse_probe + content_ci) · live-verified on a fresh boot. Status: **shipped, complete_pending_review.** Only Aevi closes.

Erik, v1.8.42: *"I thought GM corrections was already done? I tried to get the GM to do it. It wouldn't."* And — in the very window this shipped — Aevi had to **hand-repair Silas Weir with a git commit** (`e02b9dd`) precisely because *no repair UI existed*. That is the symptom this closes: a player should never need a language-model argument OR an Aevi commit to fix a bug the game gave them.

## Root cause (confirmed at the substrate)
Erik's request routes through **`[to the GM]` → `gmAsk`**, the out-of-character meta channel. That path's own system prompt says *"the story does not advance… nothing in the world changes,"* and it returns **only `{ok, text}` — it never emits or processes `stateOps`.** So the GM's refusal wasn't stubbornness; the meta channel is **structurally incapable** of repairing anything. SNG-070's `applyStateOps` only runs on the in-fiction turn. The real fix is therefore **not** more prompt-nudging — it's a direct UI. (Defect B was right.)

## Defect B — the Repair panel (the real fix)
A **🔧 Repair character** button on the Character screen opens `renderRepairScreen()`, which exposes `engine/corrections.js` (`applyStateOps`) **directly** — same engine, same guardrails, same append-only ledger as the GM's in-fiction repair. **The UI cannot do more than the GM can; it just cannot argue.**
- **Origin / people** (all 27, from `origins.json`) · **Background** (categorized) · **Form** (free text; a change re-mints the portrait, mirroring the sheet's form editor).
- **Domains** — three ring-ordered selects (primary/secondary/tertiary, + "none"). Copy is explicit that changing a domain **grandfathers held abilities and only gates future learning** — it does not strip or grant.
- **Strip an ability you never chose** — a checkbox per held ability with a live `breadth used / cap` counter; copy states plainly that a strip **frees breadth, never touches skill points, is not a loss of earned power.**
- **Remove an unmet companion** · **Unstick a quest** (force to active/available/resolved/failed) — sections render **only when the character has any**, so the panel never shows an empty control.
- Ops are **chunked by 6** (the engine's own per-call cap) so a large repair applies fully. Applied + refused are both surfaced; the **correction log is rendered** ("a repair is a fact about your character, not a secret").
- **Repair, not wish** is stated twice: the engine already refuses every advancing op, and the UI offers no control that could request one (dropdowns list only real, valid values — refusal is impossible by construction, not just by check).

## Defect A — GM behaviour (the in-fiction path + the meta channel)
- **`gmAsk`** now **welcomes repair asks and points them to the panel** ("Open Character → 🔧 Repair character…") instead of treating `[to the GM]` as a jailbreak and flat-refusing. It may confirm what a correct fix looks like, but the doing is the player's in the panel — it never promises to edit the sheet from a channel that can't.
- **Turn-path rule 63** gains **DO THE PART YOU CAN**: never blanket-refuse a mixed ask. On "replace these abilities," strip the wrong ones (a repair) and refuse the grant (an advance), keep the ones the player names, and tell them their breadth is freed and their points are theirs. *A blanket refusal where a partial repair was available is the worse failure.*
- `corrections.js`: `describeCorrection` article fix ("an ability").

## spec_boundaries
- The panel does not offer **`reanchorLocation`** or **`fixCodexFact`** (both exist in `applyStateOps`) — reanchoring wants a location picker and fixing a codex fact wants the codex UI; both are cheap follow-ons if wanted. Domains/background/origin/form/strip/companion/quest cover the reported cases.
- **Reversibility is logged but not yet one-click.** Corrections are append-only and fully recorded (from → to), so an undo is a small follow-on, but the panel currently shows the log without an "undo this" affordance.
- Origin correction validates against traditions (all 27 origin ids are tradition ids, so every offered origin passes) — a future non-tradition origin id would need the engine's origin whitelist widened.
