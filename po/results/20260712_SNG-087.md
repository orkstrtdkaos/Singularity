# Results — SNG-087 (cross-device discovery: a new device finds your characters)

Date: 2026-07-12 · v1.8.45 · npm test green (smoke + parse_probe + content_ci) · verified end-to-end live (fresh device, GitHub API stubbed). Status: **shipped, complete_pending_review.** Only Aevi closes.

Erik: *"will I be able to play Silas on my phone? without importing?"* — today: no. This wires the last mile of the BATCH-7 P2 promise: sync could REFRESH a character a device already had, but a fresh device with empty localStorage showed an empty roster and had no way to reach the characters living in the repo. Every piece existed (`ghList`, `fetchRepoJSON`, `fetchRemoteCharacter`, `adoptRemoteCharacter`, `resolveSaveConflict`); the discovery flow that strings them together did not.

## What was built (app.js only — no engine or content changes)
- **`renderDiscover()`** — `ghList("players/")` → fetch each `players/{key}/profile.json` for a display name → the person picks which player they are. **Auto-resolves** when there's a single player, and highlights the one whose name matches this device's current profile. Surfaces **both duplicate "Erik" profiles** clearly (name + key), so even before the origin-merge the pick is unambiguous.
- **`renderDiscoverCharacters(entry)`** — becomes that player on this device (stores the profile via `saveProfile`, sets the active `playerKey`, `loadIdentity`) → `ghList("characters/{key}/")` → fetches each character → **Adopt & play**. `adoptRemoteCharacter` writes it local preserving the remote's timestamps; the **stale-overwrite guard (`resolveSaveConflict`) runs in BOTH directions** — a newer local is never clobbered, a newer remote is always taken, and a genuine both-advanced conflict preserves the loser via `preserveRecovery`. Each character is marked **"already here"** when a local copy exists.
- **Three entry points** (sync config is the only setup a new device needs — no export/import):
  1. **Auto-run once** when the roster is empty and sync is on (`_discoverAutoRan` guards against re-bouncing so "Back" lands cleanly).
  2. A **"☁ Find my characters"** button on the roster (for a device that has some but not all).
  3. **"☁ Find me in the shared world"** on the player-pick — the truly-fresh-device case with no local player at all.
- **Hardening:** `renderRoster` now guards against a null `profile` (a fresh device can reach chrome's nav before choosing a player) and redirects to the player-pick; `renderDiscover`'s Back targets the player-pick when no profile exists yet.

## spec_boundaries
- **The duplicate-Erik profiles are NOT merged at origin (SNG-045 Part A).** That is a destructive data reconcile of remote profiles (union the characters, keep one canonical key, redirect the other) — I did not run it unprompted, since a bad merge could lose data. Discovery **mitigates** it (both Eriks are shown clearly and you pick the one with your characters), but the clean fix is still owed: it wants an explicit, backed-up merge op (surface both, union, redirect) — recommend Aevi authors the reconcile as its own task, or confirms CCode should build a guarded `mergePlayers(fromKey → toKey)` with a backup-first + preserve-recovery discipline.
- **The repo write-back on adopt** re-stamps `updatedAt` (via the final `saveCharacter`), matching the existing Play-button path's convention (app.js:817) — adoption reads as a light local touch. Benign (next push re-uploads identical content with a newer stamp); flagged for parity awareness.
- Discovery lists are **not paginated** — GitHub's contents API returns up to 1000 entries per dir, far above any real player/character count; noted rather than handled.
