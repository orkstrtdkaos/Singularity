# Results — SNG-091 (the player's plan request is unconditional — decouple gambitOps from gambitApt)

Date: 2026-07-13 · v1.8.47 · npm test green (smoke + parse_probe + content_ci) · verified live end-to-end. Status: **shipped, complete_pending_review.** Only Aevi closes.

Erik (v1.8.46 leg): *"I put the text in but it didn't start the gambit screen automatically… when I clicked the plan button it DID autopopulate."* SNG-088's wiring was correct and complete; the **prompt** strangled it, and two instructions fought.

## Root cause (Aevi's spec, as the ticket diagnosed)
`gm.js` gated `gambitOps` on *"a **gambit-apt** objective"* — but `gambitApt` says **OMIT IT ALMOST ALWAYS… most turns: no gambitApt.** So the GM checked aptness, concluded (as instructed) that the scene almost never is, and declined to open the board. It obeyed both rules perfectly and did the wrong thing. **`gambitApt` is the game OFFERING a plan unprompted (must be rare); `gambitOps` is the PLAYER ASKING (must be unconditional).** The player's request was chained to a heuristic built to suppress the game's suggestion.

## The fixes
1. **Decoupled `gambitOps` from `gambitApt`** (`gm.js`). Rewrote the instruction: emit `gambitOps` when the player asks to plan **IN ANY SCENE, APT OR NOT** — *"⛔ DO NOT JUDGE WHETHER THE SCENE DESERVES A GAMBIT — the player has already judged; aptness is IRRELEVANT to this op."* Added the explicit contrast (gambitApt = game offering, rare; gambitOps = player asking, unconditional — "a quiet conversation with no obstacle at all still opens the board the moment the player asks to plan"). Deleted every "gambit-apt objective" reference; changed "Omit gambitOps on every ordinary turn" → "Omit gambitOps ONLY when the player did not ask to plan."
2. **Stopped the silent drop** (`app.js`). Both `applyTurn` and the `renderPlay` guard were gated on `!gambitDraft`, so a lingering stale draft (e.g. from an abandoned build) silently swallowed the request — *"a dropped request is invisible and looks exactly like the GM ignoring you."* Now the request is honored regardless: it opens the builder, **replacing** any leftover draft, with a transparent note — *"Your previous unsaved draft was replaced — ↺ Start over for a blank plan."*
3. **`gambitApt` unchanged** — it remains rare, suppressed, unprompted-only. That heuristic was correct; it was only wrong as a *gate on the player's own request*.

## Verification (live, end-to-end)
Set up a scene with **no obstacle at all** (an ordinary "planning over tea" beat) **and a stale gambit draft lingering** (the array-office plan left by the dev builder). Submitted **"I want to plan this out"** as a freeform action → the GM emitted `gambitOps` → **the builder OPENED** pre-filled with the fresh plan (goal *"plan a quiet supper for Pell's name-day"* + 2 steps), **replacing** the stale draft, with the note *"…Your previous unsaved draft was replaced — ↺ Start over for a blank plan."* No console errors.

**Bonus finding — the SNG-088 live-test misses are explained.** `gmTurn` rejects any turn without a `choices` array (`BAD_SHAPE` → salvage path, which stored the raw JSON as narration and dropped top-level ops). The SNG-088 Part B stub omitted `turn.choices`; a real GM turn always includes it. With a `choices`-valid turn, `applyTurn`+`renderPlay` open the builder exactly as designed — **the SNG-088 wiring was correct all along; the failure was the test stub, not the product.** (Also confirmed directly: `callClaude`+`parseLooseJSON` parse a `gambitOps` payload cleanly, narration + gambitOps intact.)

## spec_boundaries
- The stale-draft collision is resolved by **replace-with-note** (the freshly-asked-for plan wins; the leftover is discarded with a visible message and a ↺ Start over affordance). The spec allowed "merge OR ask"; replace-with-note is the least surprising for the actual trigger (the player just asked for *this* plan while a stale/abandoned draft sat in memory) and never silently discards the request.
