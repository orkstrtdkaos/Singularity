# Results — SNG-115 + SNG-116 (two live bugs)

Date: 2026-07-16 · HEAD `51c16ce` · **v1.8.76** · `npm test` green · boot-clean. Status: **both shipped, complete_pending_review.**

## SNG-115 — Feedback submit no longer hangs on "Sending…"
ROUND-2 found the spec's network-timeout diagnosis **and a second, more immediate cause**:
- **The real hang:** the feedback entry (app.js) was created with **no top-level `at`**, but `submitFeedback` dates the file from `entry.at` *before* its try block — so it threw `Cannot read 'slice' of undefined`, the catch never ran, and the handler had no guard → the UI sat on "Sending…" forever with an unhandled rejection. **Fixed:** `at` is set on the entry; `submitFeedback` is defensive (`entry.at || entry.context?.at || now`).
- **The network timeout (spec's fix):** `ghGet`/`ghPut` now route through `ghFetch` — an `AbortController` with a **12s per-request deadline** (Q1: per-attempt, so a slow-but-progressing 409 retry isn't killed). On timeout the request is cancelled and throws `GH_TIMEOUT`, so the caller's catch runs and feedback lands in the "never lose it" queue. **Fixed at the `ghGet`/`ghPut` layer (Q2)** — character save + ledger inherit the same bounded wait.
- **The handler guard:** `fb-send` disables the button while sending, races the submit against a **15s UI deadline**, and **always lands on a terminal status** (sent / saved-will-retry), queuing the entry on any failure. Never a permanent spinner; never a lost entry.
- Exported `raceTimeout` (pure/testable). 3 smoke tests: a stalled promise rejects with the label + fires the abort within the deadline; a resolving promise passes through; a real 409 still propagates (the deadline doesn't mask genuine failures).

## SNG-116 — Difficulty preview now includes the substrate penalty
The "how hard is this" readout called `successChance` **without `substratePenalty`** (assumed a full lattice) while the real resolution applied the SNG-090 penalty — so on thin substrate it read "doable," then the roll was hard. **The readout was honest about a wrong number.**
- **New `substratePenaltyFor(choice, location)`** — the single source of truth: `usesAbility ? substrateForAction(...).chancePenalty : 0`, the exact value the resolve path applies.
- **Both preview sites** now pass it: the how-hard readout (~L5147) and the AUTO-intensity `stdChance` (~L2796, so auto-intensity sees the real chance too — Q2's other site). The resolve path already applied the same `substrateForAction().chancePenalty`, so **preview and resolve derive it from the same place and cannot drift.**
- Not a fog change — this fixes the *number*; the tier gate over it is unchanged (SNG-098/sense.js). 3 smoke tests: the penalty lowers the previewed chance by exactly itself; it's its own honest named breakdown line (SNG-106); no-op on a full lattice (no regression).
