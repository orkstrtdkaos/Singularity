# Results — Sidebar batch (SNG-120 · 121)

Date: 2026-07-16 · HEAD `16c6c18` · **v1.8.82** · `npm test` green · browser-verified. Status: **both shipped, complete_pending_review.**

Built together — SNG-120 (declutter the sidebar) and SNG-121 (pin the useful items) touch the same sidebar render and ship as one version bump. The redundancy SNG-120's spec names is real: SNG-119's place-scoped "who's here" already listed the same people as the old bare "People you know" block, so 120 deletes the latter.

## SNG-120 — Collapsible sidebar + combine redundant sections
- **Every sidebar section is now collapsible** via the native `<details class="sidebar-sec" data-sec="…">` idiom (invents nothing new). Nine keyed sections: `attributes`, `abilities`, `quests`, `company`, `items`, `whoshere`, `playstyle`, `maprest`, `codex`.
- **Open/closed state persists** to `profile.uiSidebar = {open:[], closed:[]}` — `loadSidebarState()`/`saveSidebarState()` + a `sectionOpen(key, defaultOpen)` resolver (explicit closed wins, then explicit open, else the play-serving default). A single `details.sidebar-sec[data-sec]` `ontoggle` handler writes the set back on every collapse/expand. Same open-set discipline as the existing `npcGroups` pattern.
- **Defaults serve play:** abilities / quests-when-active / company / items / who's-here open; attributes / play-style / map&rest / codex collapsed. Collapsed headers carry a summary count in `.sec-sum` (e.g. Quests `(2)`, Company `(3)`).
- **The real combine — "People you know" DELETED.** The old bare roster list (a duplicate of SNG-119's richer place-scoped section) is gone; its people now show exactly once, in the "{place} — who's here" section. The committed-partner banner moved into that same section so nothing is lost. (The string "People you know" survives only in an explanatory comment documenting the removal.)
- **Party + Companions → one "Company" section** (`data-sec="company"`, two `.company-group` blocks); the section disappears entirely when solo with no companions.
- Interactive controls kept out of `<summary>` so a click doesn't toggle the section: the Abilities "⬆ Level Up" button moved into the body; the pin button uses `e.stopPropagation()`.

## SNG-121 — Pin items to the sidebar
- **A `pinned` flag per item** (`character.inventory[i].pinned` — on the record, per Q2; travels with the item, survives reorder). The sidebar **Items** section renders `pinnedItems(character)` only, with a `(X pinned · Y total)` header count and a **`＋ N more in Inventory`** button (→ `renderInventoryScreen()`); empty-pinned shows the hint "Nothing pinned — pin items from Inventory for quick access." Nothing is hidden — the rest is one tap away and the count says so.
- **Sensible day-one defaults** — `ensurePins(character)`: on a never-pinned character, auto-pins by kind — **weapon + consumables + anything with authored `uses[]`** — and leaves bulk `misc`/`quest` unpinned (Q3: quest items default out of the *quick* set, still in Inventory). Runs once; marks `_pinsInitialized`.
- **Defaults never override a choice** — once the player pins/unpins anything (`_pinsInitialized` set by `togglePin`, or by the first `ensurePins`), defaults never touch the set again. A player who clears every pin stays cleared.
- **Toggle from the full Inventory view** — `itemCard(it, {…, showPin:true})` renders a 📌/📍 control (`data-item-pin`); the one shared `bindItemCardHandlers` (SNG-114) wires it → `togglePin` + save + re-render. Pin is an itemCard **context flag**, not a second renderer — composes with SNG-114, no drift.
- **Q1 (equipped/primary weapon):** there is no separate "equipped" concept at HEAD — a weapon is just `kind:"weapon"` — so the default pins by `kind:"weapon"` (all weapons), which is the correct behavior given the substrate. Pinned consumables keep their qty + SNG-114 "use in scene" intent path right in the sidebar (the point of pinning).
