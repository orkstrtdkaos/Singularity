## §1 · The one number worth restating: the Valley is 11 authored, not 16

Authored locations per region: **valley 11**, then quickwood 6, manifest_domain 5, unspooling 5, the_center 5, umbral_depths 5. So the Valley is roughly **2× its neighbours, not 3–5×**.

The 16 is from Erik's live save — authored **plus generated**. That does not weaken the spec; it sharpens it. **The drain is real and it is the generated ones doing it.** The `|| "valley"` default only fires on generation, so the gap between 11 authored and 16 lived is precisely the bug's footprint, and it will widen every session until §1 lands. Worth stating that way in the spec, because it predicts the fix's effect: authored counts should not move at all, and only generated places should relocate.

## §2 · SNG-166 answers

**Q1 — named-entity derivation: authored map or lookup over records?** **Authored map, and your instinct is right.** A lookup over entity records is a lookup over `CONTENT.traditionIndex`, which is the *right* data but the wrong failure mode: a tradition with no home region silently yields nothing, and you are back to a default. An authored `tradition → home region` table is ~30 lines, reviewable in one screen, and **fails loudly** because a missing row is visible in the file. One caveat: it must be content, not code — a hardcoded map in `generate.js` is a Law 2 violation and the next person will not find it.

**Q2 — where does an unaddressed location live?** **Neither a holding area nor a refuse-and-retry.** Mint it with `regionId: null` and `regionSource: "unresolved"`, and let it exist. Two reasons: a retry costs a model call for something the fiction has already narrated the player into, and a holding area is a second place to look for places. Everything downstream already tolerates a null region — `autoMapPositions` falls through to `parentId`/connections (CCODE-11), and the region tier simply will not list it. Surface it in the GM block and in the admin/dev view as *"this place has no Reach yet"*, and let the next scene that names its neighbourhood resolve it. **A wrong address is worse than a known-missing one** is exactly right, and null is how you say "missing" in a way the code already handles.

**Q3 — display-name migration cost.** **Measured: 52 prose occurrences across 18 files** (excluding your own spec, 45 across 17). The concentration is helpful — `origins.json` 11, `regions.json` 6, `reach_space_time.json` 5, `peoples_of_kind.json` 4, `tradition_profiles.json` 4; the tail is 1–2 each.

**Ids are provably safe: 0 of 95 locations carry a display-form `regionId`.** Every one uses `the_kept_reach` form, so no save, no `regionsKnown`, and no substrate key is touched.

The work is find-and-replace plus a read-through for the sentences where the name does grammatical work — *"the Kept Reach's archives"* → *"the Kept Hours' archives"* changes the possessive, and `reach_space_time.json`'s five hits are ability prose where the name may be load-bearing rather than incidental. **A couple of hours, fully reversible.** That is a number Erik can rule on rather than a risk.

**Q4 — naming block prompt weight.** **Always-carry, and it is cheap.** A per-region block of phonology + structure + 8–12 exemplars + an avoid list is ~120–150 tokens. Against a measured ~17k-token cached prefix that is under 1%. More importantly it belongs in the **world tier**, which is cached and stable per region — so it costs approximately nothing per turn after the first in a region. This is unlike the waygate block, which is conditional because it is *situational* (you are standing at a gate or you are not), not because it is heavy. Naming is always relevant; carry it.

## §3 · SNG-167 answers

**Q1 — region-lore auto-pull: always, or conditional?** **Always.** Same reasoning as naming: it is world-tier, cached, and stable while the player is in a region. The current concentration proves the cost is affordable — `the_twelve_reaches` is already carried by 80 of 95 locations, so the GM is *already* paying for a large lore file every turn. Adding one region file per region is a smaller marginal cost than what is already being carried. **The real win is that it removes a per-location authoring obligation** — 3 of 25 regions having a file is not an authoring backlog, it is a symptom of opt-in.

**Q2 — is `hooks` close enough to reuse?** **No — reuse `want` instead.** Measured: **1 of 43** NPCs has `hooks`, and its shape is a list of narrative threads *about* the character ("her deepening bond with Cellaceron"), not offerable opportunities. Overloading it would mean redefining a field with one instance, which is free, but it would also mean the name lies. Meanwhile **41 of 43 carry `want`** — which is already the arc premise, exactly as §2 says. So: add `questSeeds` as the new authored field, and make the CI check *"an NPC with a `want` and no `questSeeds`"* — that names 41 real gaps on day one, which is a content backlog with a number attached rather than a vague ask.

**Q3 — does Coliseum standing ride `standingOps` unchanged?** **Mostly, but "how you won" needs an engine path.** `standingOps` is right for the ordinary case. The problem is that §3a's best idea — *a clean loss to a Marcher earns more Marcher standing than a graceless win* — is precisely the judgement a model will get wrong in the flattering direction, because it is being asked to grade the player's conduct in a scene it just narrated. The engine already knows the facts that matter: the encounter outcome, whether a yield was offered and honoured (`lethalOfferClamp` already tracks the decline path), and the harm rung actually used. **Derive conduct from those, let `standingOps` carry the rest.** Otherwise every bout reports honourable.

**Q4 — where to decide the model tier?** **Before the call, no pre-classify needed.** `runGM` already assembles the whole context bag via `gmEnv()` before `gmTurn` — every signal §4 names is present at that point: `activeEnc()`, `awaitingResolution` on a quest, an `unlock`, the resolution's own degree. A tier function over the env bag is pure, testable, and free. **Do not pay for a Haiku pre-classify**: it adds a round-trip to every turn to save money on some turns, which is the wrong trade against the latency constraint SNG-155 already established. Escalate-when-uncertain falls out naturally — default Sonnet, and *demote* to Haiku only when no charged signal is present.

**One flag on §4:** the tier decision must be recorded on the turn and surfaced in the feedback report. If a scene reads flat, the first question is which model rendered it, and right now that would be unanswerable.

## §4 · BATCH-12 answers

**Q1 — α as a single named constant, against the SNG-078 blocker.** **Acceptable, and it is the cheapest possible shape.** The cliff is structural: 0.12 → 0.90 in one move is not a tuning disagreement, it is a missing transition. One named constant in a rules file, with the before/after already measured (worst 0.78→0.35, mean 0.287→0.170), is the *minimum* eyeballed surface for fixing it — and it is one line to retune. Erik holds the blocker, but I would not hold the cliff behind it.

**Q2 — `peopleDisposition` migration cost.** **Effectively zero, and it is additive.** It already lives on the character (`quests.js` creates it lazily with `character.peopleDisposition ||= {}`), and every reader already tolerates absence. Moving it to the shared shape means *documenting* where it already is and giving NPCs the same field. **No save migration is required** — Erik's and Brayden's characters gain an empty object on first write, exactly as today. The genuinely new work is §3b seeding, and that is a reconcile step, which is now a well-worn path.

**Q3 — the clock for passive drip.** **Per in-game day, not per session or per scene.** Per-scene couples standing to play *intensity* — a long conversational evening would out-earn a week of travel. Per-session couples it to real-world habits, which is worse: two short sessions beat one long one. The in-game day is the only clock that means the same thing to the fiction and the player, `advanceClock` already exists, and `worldDay` is already stamped on everything. It also makes the fix legible in the way §3c wants: *"Calvar has been with you eleven days"* is a sentence; *"Calvar has been with you nine scenes"* is not.

**Q4 — can `depended on by` and blast radius be generated reliably?** **Yes, and I checked specifically: `app.js` contains ZERO dynamic imports.** Every engine module is statically imported, so the graph is fully derivable from import statements across 53 modules. Two honest caveats to record in the map itself: (a) the GM-verb column cannot come from imports — it comes from the reply contract and the `applyTurn` dispatch, so it is a separate derivation; (b) `player-visible surface` is **not** derivable at all and must be authored, same as `purpose`. That is fine — those two are the columns worth a human's judgement, and everything mechanical around them is trustworthy.

**Q5 — does anything here collide with the outstanding `applyTurn` throw?** **No collision, but §3d raises the stakes.** `standingOps` adds another op to the `applyTurn` dispatch, which is the function that threw. The CCODE-07 guard means a throw there can no longer eat the narration, so the failure is contained — but until we have the stack, every new op in that function is added blind. **Not a blocker; a reason to get the report.**

## §5 · On sequencing — one challenge

§6's order is right and I would build it as written, with one change.

**§5 ENGINE_MAP should not be first in the sense of "finish it before anything starts."** The `purpose` column is 53 authored sentences and the map is most valuable when it is *complete*, but §3a–3c is the highest play-value work and Erik's standing screen is empty **today**. Doing 53 sentences before he sees standing move is the wrong order for the person waiting.

**Split it:** generate the mechanical columns first (imports, dependents, blast radius, content reads — all derivable, one script, no authoring), which is what actually prices §1 and §3. Author the `purpose` and `player-visible surface` columns incrementally, one module at a time, as each spec touches it. The CI gate lands with the generator; a module without a purpose line warns rather than fails until the backfill completes.

That preserves §5c's real argument — *know the blast radius before you change substrate.js or the character schema* — without putting an authoring pass in front of the thing Erik is actually waiting for.

## §6 · Two things I would add

**1. §1's fix has a matching guard, and it is the same one that just proved itself.** The seed guard caught 10 dead encounter seeds because it checked that a reference *resolves*. `regionSource` deserves the same: **a CI check that no location carries `regionSource: "default"`** — because if the fix is implemented as "derive, else default", the default will come back. Making the absence of a default enforceable is what stops §1 being re-fixed in three months.

**2. SNG-166 §3's evidence is thinner than the spec implies, and the conclusion survives anyway.** I checked Erik's live registry: 20 known people, **zero given-name repeats** — one Mara. The "keeps meeting Mara" is across characters and sessions, not within one save, so a within-save repetition ratchet would not have caught it. That does not weaken the fix — the argument for authored onomastics is the *absence of a system*, which is confirmed — but the **repetition ratchet in §3 should count across all characters on the device**, not per-character, or it will measure the wrong thing and read green.
