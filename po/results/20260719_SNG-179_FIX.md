## §1 · The fix: read the bond you already have

`bondType: "mentor"` on an `npcUpdate` **is** a teaching relationship. The engine now derives the teacher from it — the mentor's own primary domain gives the tradition, `willing:true` because a mentor is willing, the NPC id carried through. An explicit `markTeacher` still works; this is additive.

Your framing was exactly the design: *"read it as one rather than asking for a second declaration."* The diagnosis proved the model records the mentoring richly and never fires the thin one-shot beside it, because **asking it to say the same thing twice is asking it to choose**, and it chooses the expressive op. Removing the second ask removes the choice.

**It never invents.** A mentor with no resolvable domain — a registry-only NPC with no backing record — binds *nothing*. That is honest rather than a guess, and it is tested.

## §3 · The rebalance — both rules ended with a brake

Your 3b was right and it generalised: **19C and 19B both spent their last words telling the model NOT to fire.** The closest text to the decision was a caution, and the diagnosis named that as caution-as-distortion.

Moved the cautions to the **middle** and ended each on the qualifying condition:

- **19C:** *"When the beat is a genuine lesson, mark it."*
- **19B:** *"When the listed craft carries the scene, mark it."*

**No emphasis added** (§4.3). The rule was read, understood and obeyed — louder produces over-firing and leaves the substitute in place. And the engine's own confirm-and-refuse is reframed from a *warning* ("if it isn't ripe, narrate the effort") into a *safety net that permits* ("the engine refuses an unripe one, so you cannot over-award — given that, you MUST…"). Same facts, opposite pressure at the decision point.

## §4 · ⚠️ THE VERIFICATION IS YOURS, and here is exactly what to check

You said it and you are right: *"a green unit test would be least convincing."* The unit tests assert the *mechanism* — a mentor beat binds a willing teacher, the capstone gate opens, an unattributable mentor binds nothing, a `student`/`platonic` bond does not. That is necessary and not sufficient.

**The reproduced symptom, in order:**

1. **Re-run "Ask Veth to teach you her most valued lesson"** — the exact turn that produced the null.
2. **Confirm `character.teachers` is non-empty.** This is the direct symptom that sat empty for sixteen levels.
3. **Confirm the downstream unlock opens** — that people's capstone gate is now reachable (the thing the empty `teachers` was silently blocking).

**A dependency you will hit, and it is not a fault in this fix — it is the same open gap.** The captured Veth turn set **neither `bondType` nor a domain**. Veth is `sworn`, registry-only, no backing record — so:

- **The model must set `bondType:"mentor"` on the mentor beat.** The 19C rebalance makes that likelier, but it is a model behaviour and the live turn is the only test of it. If it sets the bond, the teacher binds.
- **Veth needs a domain.** A registry-only NPC has none, so even with the bond set, there is no tradition to teach. This is the **hub-attribution gap still open** (SNG-171/177 for registry-only NPCs) — the same one behind the Ent crediting nothing. Until it closes, the derivation works for authored and generated NPCs and abstains for registry-only ones.

So the honest end-to-end status: **the mechanism is correct and shipped; Veth specifically will bind a teacher once she carries a domain and the model sets the bond.** If you re-run and `teachers` is still empty, capture the raw `npcUpdates` again — if `bondType` is set and it still didn't bind, that is a bug in my derivation; if `bondType` is absent, the rebalance needs another turn of the screw and that is prompt work, not engine.

`logOpOutcome` now records `"derived-from-bond"`, so the op-ledger on your next feedback report will show a derived teacher distinctly from an explicit one — and, for the first time, a non-zero count for an op that read zero for sixteen levels.
