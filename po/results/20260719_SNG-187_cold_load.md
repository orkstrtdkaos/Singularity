## §1 · The cause, confirmed exactly as you diagnosed

`loadContent` awaited ~250 JSON files **strictly sequentially** — every `for … of` loop did `await fetchJSON(…)` inside the loop body, so each request waited for the one before it. `Promise.all` appeared zero times. At a ~60 ms CDN round-trip, 250 serial requests is **~15 s of pure latency** before any transfer — which is the 15.30 s you measured. Not a payload problem; the bytes were always fine.

I counted the fetches to be sure: **250** (your "252 files", near exactly).

## §2 · The fix — three stretches, ~250 round-trips → a handful of waves

Every group reads a path list known in full before the first request, so the fetches have no inter-dependency and run in parallel. Three sequential stretches converted:

1. **The ~12 core rule-loads** (base rules + emergence + gates + traditions + …) → one `Promise.all`.
2. **The 10 content groups** (abilities, core items, valley items, **locations (96)**, **npcs (42)**, events, companions, encounters, lore, quests) → one `Promise.all`/`allSettled` each, and the promises are created **before the first await** so the groups overlap too.
3. **The ~14-fetch tail** (region, generative substrate + arcs + 3 gen schemas, origins, backgrounds, terrain regions, accords, helper text, substrate model, prologue, legends) → one `Promise.all`.

## §3 · The two cautions — held, and the second one *proven*

**Caution 2 — failure tolerance preserved.** Where the old code `try`-wrapped a group so one bad file didn't kill the load — **valley items** and **quests** — it now uses `Promise.allSettled` and skips the rejected. Every group that was fatal-on-miss before (abilities, core items, locations, npcs, events, companions, encounters, lore, region) stays fatal via `Promise.all`. No failure semantics changed.

**Caution 3 — order-independence, the one place this could silently change content.** `Promise.all`/`allSettled` **preserve input order** in their result arrays, so every fold loop runs in the same manifest order it did sequentially. For an id-keyed map (`abilities[a.id]`, `items[it.id]`, `locations`, `npcs`) that means the **same entry wins a duplicate-id collision** — last-write-wins is unchanged. For the `quests` array it means the same concatenation order. Valley items still overlay core items (folded after). Legends still hydrate into npcs without overwriting a same-id registry NPC.

**I did not assume this — I proved it.** A Node harness (a `fetch` shim over the real repo files) ran `loadContent` end-to-end before and after, and compared a fingerprint of every group: the counts, **plus a value-size hash per id-keyed map** (which catches a reordered collision winner, not just a dropped file), plus `abilitiesWithAccord` and `legends-in-npcs` (the two cross-mutations). Every field matched **to the digit**:

```
abilities 285 · items 30 · locations 95 · npcs 47 · challengerPools 1 · events 1 ·
companions 9 · encounters 19 · lore 27 · quests 4 · randomEncounters 58 · accord-tagged 7
abilitiesFP 5574/-2096989565 · itemsFP 527/655939958 · locationsFP -1186857652 ·
npcsFP -2069342649 · questsFP -645553368 · loreKeys 314773322      [identical both runs]
```

That run also proves `loadContent` executes end-to-end without throwing — an integration test of the exact function changed.

## §4 · The win, quantified

localhost has ~0 ms latency, so it cannot show the fix (your point exactly: "latency does not appear in a test that reads from disk"). So I injected a synthetic per-fetch round-trip and timed the parallel load:

| | |
|---|---|
| fetches | **250** |
| peak concurrency | **221** in flight at once (was 1) |
| parallel wall-time @ 25 ms/fetch | **258 ms** |
| sequential-equivalent (250 × 25 ms) | **6250 ms** |
| **speedup** | **24×** |

Scaled to the real CDN's ~60 ms round-trip, the **sequential path is ~15 s** — reproducing your measured 15.30 s and confirming the diagnosis — while the parallel path is a few waves (manifests → spectrums → rules → groups → tail), comfortably inside the **under-2 s** target.

## §5 · Not done, deliberately

- **§3.5 early paint** (something on screen before the corpus finishes) — not needed. The parallel load is fast enough that a skeleton-before-content adds complexity for a win the parallelism already delivers. If your CDN LCP still disappoints, this is the next lever; measure first.
- **§5 bundling** per-pack manifests into single payloads — **almost certainly moot now** (250 round-trips → a few waves), and a build step is a real cost. Measure the new LCP before considering it.
- **Prompt caching** — not touched (§4).
