## ROUND 2 — answered from real reads (not taken on report)

- **Q1 — new canon KIND vs own file → OWN FILE (new store `world/braid_recipes.json`).** The entity-canon store (`canon.js`) is entity-shaped: npc/location/arc keyed by entityId, engagement-promoted (nominated tier), weight-CONTESTED (`resolveContradiction`), rating-lensed on entity fields. A braid recipe is none of those — ability-def shaped, keyed by `braidKey`, found-once, with no realness axis. Shoehorning it into the entity store would drag `promoteInto`'s contest logic across a thing it doesn't fit. Own file, same `pushMergedFile` concurrency (§3.1 honoured).
- **Q2 — adoption timing → CHECK-AT-RIPEN when synced, MINT-THEN-RECONCILE offline.** Both of §3.3's needs at once: if sync is on, the mint path looks up the cached recipe BEFORE authoring — found → adopt (recognition beat, and it SKIPS the LLM call); absent → author + mint + publish (creation beat). Sync off → mint locally (creation), and the next `syncBraidRecipes` publishes as first-finder OR adopts if someone landed it meanwhile. The check rides the already-async mint; adopting is strictly cheaper than authoring.
- **Q3 — rating-lens for ability prose → PASSES THROUGH, no new machinery.** `canonForViewer`/`lensDecision` judge entity-rating tags on entity fields (role/wants/fears/appearance…). A braid recipe is a craft description authored under the player's own rating ceiling at authoring time (same as all generation); it carries no entity-rating tag, so the entity lens has nothing to act on. Recipes are not run through it.
- **Q4 / §3.5 — emergence_recipes reuse → CONFIRMED PRESCRIPTIVE, NEW STORE.** Traced every consumer of `CONTENT.emergence`: `ripeCombos`/`ripeBranches`/`validEmergenceId` (practice.js) GATE what authored combos/branches a character may CLAIM (`acceptCombo`/`acceptBranch`). Putting recipes in that file risks a consumer reading them as claimable gates — the exact SNG-196 bug reborn. New store, zero overlap. The recipe stays DESCRIPTIVE (what a pairing became), never prescriptive (what it's allowed to become).

**§3.2 divergence (flagged):** recipes need **no rank-by-realness contest**. A recipe has no realness axis (unlike a contested world-FACT), so "first finder = first to land a PUT" is the honest, deterministic resolution — and `pushMergedFile`'s SHA-retry already gives it for free (`mergeRecipes` is first-PUT-wins, recipes immutable once landed). The loser's concurrent PUT re-merges, sees the landed recipe, and their local braid adopts it; their authored name becomes their personal nickname (local only). This is simpler and more deterministic than importing the entity weight-roll. If you want the weight-roll anyway, say so — but I think a craft-definition contest by realness would be inventing a dimension recipes don't have.

## The build

**`engine/recipes.js` (new, pure, tested):** `ensureRecipeStore`, `buildRecipeRecord` (STUB never promotes — §1: the failure fallback is never the world's recipe; and NO numbers travel — tier/levelReq/energy are absent), `recipeFor` (by pairing, order-independent), `recipeToAuthored` (adoption runs the SAME validated mint path — the emergent verb is re-checked against the current vocab on adopt), `mergeRecipes` (first-PUT-wins), `firstFinderName`.

**`app.js`:** `authorAndMintBraid` now adopts-at-mint when `braidRecipeStore` knows the pairing (recognition moment via `def._recognition`, no LLM call) or authors fresh as first-finder (creation moment). `syncBraidRecipes` (hooked beside `syncSharedCanon` in the sync cadence) publishes first-finder braids + adopts any landed meanwhile, riding `pushMergedFile`; refreshes the cache for the mint-time lookup. `adoptRecipeOntoLocal` rebuilds a local def from the recipe keeping the ADOPTER's numbers; a player nickname (`namedBy: player`) renders locally and is never overwritten by the world-name (§2 ratified). The moment modal reads recognition-vs-creation and shows "first found by <name> — and now yours too".
