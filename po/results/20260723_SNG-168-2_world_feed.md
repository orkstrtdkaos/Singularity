# SNG-168 §2 — the world feed (post a turn you love to the family; never canon)

**CCode · 2026-07-23 · v1.8.242 (`c2744f87`) · suite + wiring-audit green · engine unit-proven, clean
fresh-port boot.** `status: complete_pending_review`

Erik/Brooklyn's ask: *"Post a turn you really like, with its image, so other players see it — a world-news
page that makes the world feel alive, and shares the experience with friends."* Scoped per §6: **just the
feed** (map §1 + messaging §3 decoupled), **in-app per family group**, riding the **existing family-sync
substrate** — so no new backend, no new auth.

## What shipped
- **`engine/feed.js` (new, pure):**
  - `buildFeedPost` — a post from a turn: narration (or a poster-trimmed excerpt), the turn's image, character,
    place, world-date, and the poster's **RATING** (the lens key, stamped like a canon record).
  - `appendFeedPost` — the `pushMergedFile` merge body: **idempotent** (one turn → one post), capped to the
    most recent 200, and merge-safe (re-runs against the fresh remote so concurrent writers never clobber).
  - `feedForViewer` — the reader's view: reverse-chron, **RATING-LENSED on read** with the *same* lens shared
    canon uses (`lensDecision`). A post above the viewer's ceiling is **softened + its image withheld** ("adapt")
    or **hidden entirely** ("filter", e.g. sexual / minor-gore). Nothing above the reader's rating ever leaks.
- **`app.js`:**
  - A per-turn **"📮 Post to feed"** control (sync-gated). The per-post consent **is** the click + a confirm —
    the value is the *choosing* (a moment you loved), never an automatic log.
  - A **"📮 Feed"** nav on the play screen → `renderFeed()`, which pulls the one shared file and lenses it.
  - `postTurnToFeed` appends via `pushMergedFile` to the shared feed; `renderFeed` reads via `fetchRepoJSON`.

## ⛔ The load-bearing guard
**A feed post is NOT canon.** It lives in its own file (`world/feed.json`) and **never hydrates into another
player's CONTENT** — that is the separate shared-canon path, with its own reconciliation. The feed is a
read-only scrapbook. Keeping these apart is the whole reason §2 was specced beside shared-canon (§2 "What it
must NOT do") — they look mergeable and are not.

## §6 decisions honoured
- **Where it lives:** in-app, one shared file on the existing global sync repo (sync is global — one valley =
  the family). No published surface, no new auth.
- **Scope:** feed only. Map §1 (mobile pinch/pan) and messaging §3 stay decoupled/deferred.
- **Substrate reused:** `pushMergedFile`/`fetchRepoJSON` (sync), `lensDecision` (the rating lens),
  `ratingCeilingNow` (the poster's ceiling), `momentArt` (the turn image). No new backend.

## Deferred (per §6 item 4)
**World-news items** (a world-tick of consequence appearing as a feed post) — kept as a fast-follow; the
player-post feed is what Brooklyn wants first. `buildFeedPost` already takes a `kind:"world"` so a worldtick
hook can post one later with no schema change.

## Honest bound
Tier-1 (Node suite: `buildFeedPost` carries the fields + rating; empty turn → no post; append idempotent +
merge-safe; `feedForViewer` reverse-chron; a post above ceiling is lensed/softened with its image withheld;
the guard — FEED_PATH separate; the app wiring — post control, feed nav, `pushMergedFile` append, `renderFeed`
lens) + a clean fresh-port boot (the new module loads). **The live post→see-it flow is sync-gated** — the dev
preview has no PAT, so the Post button and Feed screen only appear in a synced session. That's the
**Erik/Brooklyn browser-leg Tier-2 confirm**: Brooklyn posts a turn she loves (with its image); it appears in
the family feed; Erik (or another family character) sees it in-app, rating-lensed, **without it becoming canon
